# Knowledge Capture v2 Pattern

Huginn's **Knowledge Capture v2** provides intelligent knowledge management with **semantic search**, **graph linking**, **revision tracking**, and **category-based consolidation**. Uses **agent tools** for automatic capture, **RequestContext wiring** for account isolation, and **prompt engineering** for consolidation-first behavior.

## Architecture Overview

**Four-layer system with embeddings and graph:**

1. **Data Layer:** `NoteStore` service with PostgreSQL + pgvector
2. **Embedding Layer:** OpenRouter text-embedding-3-small (1536 dimensions)
3. **Agent Layer:** Unified `capture-knowledge` tool + `recall-notes` + `delete-note`
4. **Web Layer:** Dashboard with full-text search, semantic search, and graph visualization

**Cross-channel consistency:** Same `accountId`, different `channel` values, unified `NoteSource` tracking with revision history.

<Callout type="info">
**Key v2 Change:** Category-based consolidation replaces similarity thresholds. The agent MUST update existing notes on the same topic rather than creating duplicates.
</Callout>

## Agent Tool Design Patterns v2

### 1. capture-knowledge Tool (Replaces save-note)

**Enhanced schema with linking and updates:**

```typescript
// Tool input schema
{
  title: string;
  content: string;
  tags: string[];
  isExplicit: boolean;          // User requested vs. proactive
  existingNoteId?: string;      // For UPDATE operations
  relatedNoteIds?: string[];    // For creating links
  relationship?: "related" | "extends" | "contradicts" | "supersedes";
  reason?: string;              // Required when updating
}
```

**Consolidation workflow from RequestContext:**

```typescript
// apps/agent/src/mastra/tools/capture-knowledge.ts
if (input.existingNoteId) {
  // UPDATE existing note
  const updated = await noteStore.update(input.existingNoteId, accountId, {
    title: input.title,
    content: input.content,
    tags: input.tags,
    reason: input.reason ?? "Updated with new information",
    // This automatically pushes old state to revisions[]
  });
  action = "updated";
} else {
  // CREATE new note
  const created = await noteStore.create(accountId, {
    title: input.title,
    content: input.content,
    tags: input.tags,
    capturedBy: input.isExplicit ? "user" : "agent",
    source: threadId ? { threadId, channel } : undefined,
  });
  // Fire-and-forget embedding generation if OpenRouter key provided
  action = "created";
}

// Create links if relatedNoteIds provided
if (input.relatedNoteIds?.length) {
  const relationship = input.relationship ?? "related";
  for (const relatedId of input.relatedNoteIds) {
    await noteStore.link(noteId, relatedId, relationship);
  }
}
```

### 2. recall-notes Tool (Enhanced)

**Three search modes:**

```typescript
// Full-text search (PostgreSQL tsvector)
if (input.query) {
  notes = await noteStore.searchFullText(accountId, input.query, 10);
}

// Semantic search (pgvector cosine distance)
if (input.semanticQuery) {
  const scored = await noteStore.searchSemantic(accountId, input.semanticQuery, 5);
  notes = scored.map(s => ({ ...s, similarity: undefined }));
}

// Tag-based filtering (unchanged)
if (input.tags) {
  notes = await noteStore.list(accountId, { tags: input.tags });
}
```

### 3. delete-note Tool (Unchanged)

**Ownership verification built-in:**

```typescript
await noteStore.delete(input.noteId, accountId);
// NoteStore.delete() handles ownership check and cascades links
```

## Instruction Prompt Engineering v2

### Knowledge Capture Section with Consolidation Rule

**BASE_INSTRUCTIONS emphasizes search-before-save and consolidation:**

```
## Knowledge Capture

You have a knowledge base — a graph of notes that you maintain for the user.
This is their second brain. Treat it with care.

### The search-before-save rule

NEVER create a note without searching first. Always call recall-notes before
capture-knowledge. This is the most important rule. Duplicates destroy the
knowledge graph.

Decision flow:
1. Detect knowledge worth saving (or receive explicit instruction)
2. Call recall-notes with a query describing the knowledge
3. Examine the results:

**UPDATE or CREATE? The consolidation rule:**

If recall-notes returns ANY note that covers the same category or topic,
you MUST update that note instead of creating a new one. Consolidate
the new information into the existing note.

What counts as "same category":
- Both about the user's preferences for the same kind of thing
  (languages, tools, foods, hobbies)
- Both about the same person, project, or system
- Both about the same concept, just with new details

Examples of CORRECT behavior:
- Note "Favorite language is Rust" exists → user says "I also love
  TypeScript" → UPDATE the note. New title: "Programming Languages".
  New content covers both Rust and TypeScript.
- Note "Project Alpha — Tech Stack" exists → user says "we added Redis
  to Project Alpha" → UPDATE the note with the new detail.

Examples of WRONG behavior (never do this):
- Creating "Programming Language Preference - Rust" AND "Programming
  Language Preference - TypeScript" as separate notes. These are the
  same category and MUST be one note.
- Creating a new note for every individual fact. Consolidate.

### Explicit capture
When the user says "remember this", "save this", "note this down", or
similar — save a note immediately using capture-knowledge with 
isExplicit: true. Confirm briefly: "Saved." Don't over-explain.

### Proactive capture
When a conversation contains a decision, preference, important fact, or
reusable reference that the user would likely want to recall later —
save it proactively with isExplicit: false. Keep the title concise and
the content factual. Mention it briefly: "I noted that down."

### Guardrails
- Never save transient chit-chat, greetings, or trivial small talk.
- Never save sensitive data (passwords, tokens, secrets) as notes.
- Prefer consolidation: If in doubt whether to merge or split, merge.
```

## Revision Tracking Pattern

**Automatic history preservation:**

Every `update()` operation pushes the previous state to the `revisions[]` array before applying changes:

```typescript
interface NoteRevision {
  title: string;
  content: string;
  reason: string;
  updatedAt: string;
}
```

**Implementation in NoteStore:**

```typescript
// packages/shared/src/services/note-store.ts - update method
const current = await this.get(noteId, accountId);
if (!current) return null;

// Push current state to revisions
const revision: NoteRevision = {
  title: current.title,
  content: current.content,
  reason: updates.reason ?? "Updated",
  updatedAt: new Date().toISOString(),
};

const newRevisions = [...current.revisions, revision];

// Update with new content and revisions
await db.update(notes)
  .set({ 
    title: updates.title, 
    content: updates.content, 
    revisions: newRevisions,
    updatedAt: new Date() 
  })
  .where(and(eq(notes.id, noteId), eq(notes.accountId, accountId)));
```

<Callout type="warn">
The `reason` field is required when updating notes to maintain meaningful revision history.
</Callout>

## Cross-Channel Architecture (Enhanced)

**Channel isolation with unified embeddings:**

| Channel   | ThreadId Format         | NoteSource Pattern | Embedding Scope |
|-----------|-------------------------|-------------------|-----------------|
| Web App   | `chat-${accountId}-${timestamp}` | `{ threadId, channel: "web" }` | Account-wide |
| Telegram  | `tg-chat-${chatId}`     | `{ threadId, channel: "telegram" }` | Account-wide |

**Embedding generation is channel-agnostic** — all notes for an account share the same vector space for semantic search.


### Behavioral Pattern

1. **Detection:** Agent identifies save-worthy content in conversation
2. **Deduplication:** Quick `recall-notes` check for existing content
3. **Save decision:** Explicit (user request) vs. proactive (agent judgment)
4. **Confirmation:** Brief acknowledgment without over-explaining

### Thread-Level Tracking

**ThreadId patterns:**

- **Web:** `chat-${accountId}-${timestamp}` — new thread per conversation
- **Telegram:** `tg-chat-${chatId}` — persistent chat context

**Source tracking enables:**

- Note provenance (which conversation created this?)
- Context reconstruction (what was discussed when this was saved?)
- Cross-channel search (notes from any conversation channel)
