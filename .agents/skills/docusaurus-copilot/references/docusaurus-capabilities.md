# Docusaurus capabilities reference

Use this reference when deciding which Docusaurus features to reach for.

## Verify first

Before changing a Docusaurus site, inspect the local config and structure:

- `docusaurus.config.*`
- `sidebars.*`
- `docs/`, `blog/`, `src/pages/`, `src/components/`, `src/theme/`, `src/css/`, `static/`
- `i18n/`, `versioned_docs/`, `versioned_sidebars/`

If local config is insufficient, verify current docs at `https://docusaurus.io/docs`.

## Core content surfaces

### Docs plugin

Use for structured documentation.

Key capabilities:

- sidebars and autogenerated categories
- doc front matter for SEO, pagination, tags, slugs, TOC control, draft/unlisted state
- versions and docs version dropdowns
- generated indices and tags
- edit links, breadcrumbs, last updated metadata
- multiple plugin instances for multi-product docs

Important notes:

- `routeBasePath: '/'` enables docs-only mode
- `sidebarPath: false` disables sidebars
- `editUrl` can be a function
- use file-path links so Docusaurus rewrites them correctly across versions

Relevant docs:

- Introduction: `https://docusaurus.io/docs/docs-introduction`
- Docs plugin API: `https://docusaurus.io/docs/api/plugins/@docusaurus/plugin-content-docs`
- Sidebar: `https://docusaurus.io/docs/sidebar`
- Versioning: `https://docusaurus.io/docs/versioning`

### Blog plugin

Use for releases, changelogs, engineering stories, and author-driven content.

Key capabilities:

- authors file and author pages
- tags and tags file
- feeds (RSS / Atom / JSON)
- reading time customization
- archive and pagination
- multiple blog instances
- production-friendly release storytelling

Relevant docs:

- Blog guide: `https://docusaurus.io/docs/blog`
- Blog plugin API: `https://docusaurus.io/docs/api/plugins/@docusaurus/plugin-content-blog`

### Pages plugin

Use for landing pages and custom experiences.

Key capabilities:

- React pages under `src/pages`
- Markdown / MDX pages with page-level front matter
- custom routes and wrapper classes
- ideal for docs homepages, comparison pages, pricing-style education pages, interactive guides

Relevant docs:

- Pages plugin API: `https://docusaurus.io/docs/api/plugins/@docusaurus/plugin-content-pages`

## Authoring features

### MDX and React

Use MDX for interactive docs.

Key capabilities:

- inline JSX inside docs
- imported components from `@site/src/components`
- global MDX component scope via `src/theme/MDXComponents`
- import markdown partials and raw code snippets
- access `frontMatter`, `toc`, and `contentTitle`

Important notes:

- MDX v3 is strict
- use JSX syntax, not raw HTML assumptions
- prefer separate component files when logic gets complex
- use `@site` imports instead of fragile relative paths

Relevant docs:

- `https://docusaurus.io/docs/markdown-features`
- `https://docusaurus.io/docs/markdown-features/react`

### Tabs

Best for:

- OS-specific instructions
- package manager choices
- language/framework variants
- side-by-side conceptual comparisons

Key capabilities:

- `groupId` to sync tab choices via localStorage
- `queryString` to persist choices in sharable URLs
- `lazy` rendering for performance
- custom tab heading attributes and styles

Relevant docs:

- `https://docusaurus.io/docs/markdown-features/tabs`

### Code blocks

Best for:

- polished code samples
- installation flows
- annotated diffs and walkthroughs
- interactive demos

Key capabilities:

- titles
- line highlighting via comments or metastrings
- line numbers
- custom Prism themes and extra languages
- React Live editor via `@docusaurus/theme-live-codeblock`
- `@docusaurus/remark-plugin-npm2yarn` for npm/yarn/pnpm/bun switching

Relevant docs:

- `https://docusaurus.io/docs/markdown-features/code-blocks`

### Admonitions

Best for:

- warnings, tips, trade-offs, migration notes, decision points
- nested progressive disclosure
- richer inline guidance with MDX inside admonitions

Key capabilities:

- titles with Markdown
- nested admonitions
- JSX usage
- custom admonition types and rendering

Relevant docs:

- `https://docusaurus.io/docs/markdown-features/admonitions`

### TOC and headings

Best for:

- long conceptual pages
- reference pages with stable deep links
- guided jump navigation

Key capabilities:

- explicit heading IDs
- page-level and global TOC depth control
- inline TOC with `TOCInline`
- filtered/custom TOC generation from the `toc` global

Relevant docs:

- `https://docusaurus.io/docs/markdown-features/toc`

### Diagrams and math

Use when diagrams or formulas increase clarity faster than prose.

Key capabilities:

- Mermaid via `@docusaurus/theme-mermaid`
- themed light/dark diagrams
- dynamic Mermaid component
- KaTeX math via `remark-math` + `rehype-katex`

Relevant docs:

- Diagrams: `https://docusaurus.io/docs/markdown-features/diagrams`
- Math: `https://docusaurus.io/docs/markdown-features/math-equations`

## Search and discovery

### Search

Options:

- Algolia DocSearch (first-class support)
- Typesense DocSearch
- community local search
- custom swizzled search bar

Key capabilities:

- contextual search by version and locale
- Ask AI in DocSearch
- custom search styling
- path replacement for multi-base deployments

Relevant docs:

- `https://docusaurus.io/docs/search`

## Navigation and structure

### Sidebars

Useful for:

- ordered journeys
- concept/task/reference grouping
- generated category landing pages
- multi-sidebar architectures

Key capabilities:

- autogenerated sidebars
- custom categories, docs, and external links
- custom props and class names on sidebar items
- hideable sidebar and auto-collapse behavior

Relevant docs:

- `https://docusaurus.io/docs/sidebar`

### Versioning

Use only when there are truly multiple active doc sets.

Key capabilities:

- cut versions from current docs
- control current vs latest behavior
- version badges/banners
- deploy subset of versions
- version-aware navbar items

Important notes:

- keep active versions small
- prefer absolute `@site` imports in docs components
- decide whether assets are global or versioned

Relevant docs:

- `https://docusaurus.io/docs/versioning`

### i18n

Use when there is a real translation pipeline.

Key capabilities:

- locale-specific content folders by plugin
- JSON label translations and markdown content translations
- localized assets and theme text
- scalable build/deploy strategies

Relevant docs:

- `https://docusaurus.io/docs/i18n/introduction`

## Styling and advanced customization

### Styling and layout

Prefer these first:

- Infima CSS variables
- theme class names
- CSS modules
- dark mode tokens
- mobile breakpoints
- data attributes for alternate embedded modes

Relevant docs:

- `https://docusaurus.io/docs/styling-layout`

### Static assets

Important rules:

- never hardcode base URLs in JSX
- in JSX, prefer imports / `require()` / `useBaseUrl`
- in Markdown, use Markdown syntax for asset references
- `static/` files can be copied from multiple configured directories

Relevant docs:

- `https://docusaurus.io/docs/static-assets`

### Swizzling

Use for deeper theme customization.

Safety order:

1. CSS
2. theme config
3. wrapping safe components
4. ejecting when justified

Important notes:

- wrapping is safer than ejecting
- restart dev server after swizzling
- document why a swizzle exists
- after upgrades, re-swizzle and compare diffs if needed

Relevant docs:

- `https://docusaurus.io/docs/swizzling`
